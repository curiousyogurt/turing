 #lang turing package: turing

1 Introduction

This is a domain-specific language modelled on the language Alan Turing uses in his classic 1936 paper where he introduces Turing Machines. There are many Turing Machine implementations available, but none model the language that Turing uses very closely. This implementation for Racket tries to get very close.

2 Background

In 1936, Alan Turing published a paper* that introduced Turing Machines (which he called "a-machines"). A Turing Machine is a theoretical machine that begins with a tape divided up into an infinite number of squares. The machine has a read/write head positioned over some square, and the head may read the character on the tape at that square, write a character to the tape at that square, or erase a character on the tape at that square. Moreover, the head can move one square to the left or right. The machine is always in one of a finite number of discrete states (called "m-configurations" or "machine configurations" or sometimes just "configurations").

The behaviour of the Turing Machine is described in a table of instructions, which today we would call a "computer program." Here is an example of a table of instructions (from Turing 1936, 233):

Configuration               Behaviour

      m-config.      symbol     operations   final m-config.

    b           None         P0, R            c

    c           None           R              e

    e           None         P1, R            f

    f           None           R              b

This table of instructions, when started with a blank tape and a Turing Machine in the b configuration, will instruct the machine to print 0 and 1 left-to-right continuously across the tape in alternating squares.

In order to study the computational model of Turing Machines, it is advantageous to have a language that closely models the language in Turing’s paper. Since the machines Turing describes have an infinite tape and were (for the most part) intended to run in infinite loops, any implementation requires some way to deal with the finitude of real-life computers. Additionally, it is desirable to have a way to specify the initial state of the tape as well as the position of the head and starting configuration.

[*] Turing, Alan M. 1936. "On Computable Numbers, with an Application to the Entscheidungsproblem," Proceedings of the London Mathematical Society, Series 2, Vol. XLII, 230–65.

3 Syntax and Semantics

A program expressed in the turing language has a header that specifies initialisation conditions as well as control over the operations of the table of instructions. A program also has a body that encodes some table of instructions.

3.1 Header

The header always consists of three lines, and should be present at the beginning of the code. The format of the header is as follows:
#lang turing
<tape>
<start-config> <start-head> <start-print> <end-print>

The first line, #lang turing, tells Racket you are using the turing domain specific language. This is not strictly necessary if the language is set in DrRacket.

The second line, <tape>, is a literal string that specifies the characters on the tape prior to entering <start-config> and carrying out operations. Use _ (underscore) to represent a blank square on the tape.

The third line has the following parameters:

<start-config> <start-head> <start-print> <end-print>

<start-config>: A string representing the initial configuration of the machine.

<start-head>: An integer representing the square the head starts on (where 0 is the first square, 1 is one square to the right of the first square, 2 is two squares to the right of the first square, and so on).

<start-print>: An integer representing the step in the computation (a step is a single operation) at which Racket will start printing complete configurations.

<end-print>: An integer representing the step in the computation at which Racket will stop printing complete configurations.

By specifying the characters on the tape with <tape>, the initial configuration of the machine with <start-config>, and the initial position of the head on the tape with <start-head>, the complete configuration of any machine at any point may be reconstructed (Turing 1936, 232).

The value for <start-print> must be less than or equal to the value for <end-print>. <start-print> is useful because it takes time for Racket to print successive complete configurations. If some complete configurations do not need to be printed, the computation may be carried out more quickly by not printing out every complete configuration. <end-print> forces the computation to stop, but the computation will do so earlier if the Turing Machine ever enters into the halt configuration.

3.1.1 Header Examples

Example 1

Start with a blank tape, in configuration b, and have the head start on square 0. Print output for steps 0 to 20.

#lang turing
_
b 0 0 20

The second line of Example 1 indicates that the tape is blank. The third line indicates that the Turing Machine will start in configuration b (first argument), with the head at position 0 (second argument). Furthermore, Racket will print complete configurations starting at step 0 of the computation (third argument), and halt at step 20 (fourth argument).

Example 2

Start with a tape with the characters əə0_0, in configuration begin, and have the head start on square 2. Print output for steps 400 to 500.

#lang turing
əə0_0
begin 2 400 500

The second line of Example 2 indicates that the tape starts with the characters əə0_0 written on the tape, where the first character of this string is square 0. The third line indicates that the Turing Machine will start in configuration begin (first argument), with the head at position 2, which is the third square from the left (second argument). Furthermore, Racket will print complete configurations starting at step 400 of the computation (third argument), and halt at step 500 (fourth argument).

3.2 Body

The body encodes some table of instructions for a Turing Machine. The format for each line of the body is as follows (where arguments are separated by a space):

<m-config> <symbol> <ops> <final m-config>

<m-config>: A string representing the machine configuration that, together with <symbol>, comprises the sufficient condition to execute the operations on this line.

<symbol>: A single character that, together with <m-config>, comprises the sufficient condition to execute the operations on this line.

<ops>: A string of comma-delimited characters representing the operations that the machine carries out if the machine is in <m-config> and the symbol under the head is <symbol>. Note that because space separates arguments in the line, spaces are not permitted in <ops>.

<final m-config>: The machine configuration assumed after the operations are carried out.

The arguments <m-config> and <symbol> function as a sufficient condition for executing the <ops> and the machine going into configuration <final m-config>. If <m-config> <symbol>, is not unique in the table of instructions, only the first occurrence of the instruction is acted upon.

3.2.1 Configurations

Configurations are strings of symbols without spaces. The only special configuration is halt, which may be put as a final <m-config> on a line in the body. If the machine ever goes into the halt configuration, it halts immediately. When the machine stops due to going into the halt configuration (as opposed to reaching step <stop-print> as specified in the header), Racket prints halt as the final line of the output. There is no equivalent to the halt configuration in Turing’s paper.

3.2.2 Symbols

Valid symbols are any single character, with two exceptions of symbols that have special meanings: * and _ (underscore). The * symbol matches any symbol on the tape other than _ (underscore). It is the equivalent of "Any" in Turing’s paper, which is intended to match any symbol other than a blank (Turing 1936, 234). The _ (underscore) symbol represents a blank, and corresponds to the "None" in Turing’s paper (Turing 1936, 233).

If the body has multiple matches for <symbol> with some <m-config>, precedence is given to the line which appears earlier. This may come into play when using *, since it is often desirable to match specific symbols, and then use * to catch any remaining symbols other than _ (underscore). In such a case, be sure the specific symbols are above * in the body for that configuration.

3.2.3 Operations

Valid operations are R, L, P<symbol>, E, and work as follows:

R: Move the head one square to the right of the current square.

L: Move the head one square to the left of the current square.

P<symbol>: Print <symbol> on the current square. This overwrites any symbol under the head. <symbol> is any single character. Note that P<symbol> does not automatically move the head left or right after printing.

E: Erase the symbol on the current square. This is equivalent to P_ (print underscore).

3.2.4 Body Examples

Example 3

Assuming the machine has a blank tape and starts in configuration b, print 0_1_0_1_… across the tape (Turing 1936, 233):

b _ P0,R c
c _ R e
e _ P1,R f
f _ R b

b _ P0,R c: If the machine is in configuration b, and the head is on a blank (_, underscore) square, print 0 in that square and move the head one square to the right, and go into configuration c.

c _ R e: If the machine is in configuration c, and the head is on a blank square (_, underscore), move the head one square to the right, and go into configuration e.

e _ P1,R f: If the machine is in configuration e, and the head is on a blank (_, underscore) square, print 1 in that square and move the head one square to the right, and go into configuration f.

f _ R b: If the machine is in configuration c, and the head is on a blank square (_, underscore), move the head one square to the right, and go into configuration b.

The output of a machine running this table of instructions is 0_1_0_1_…, and the head is on square 101 at step 200.

Example 4

Assuming the machine has a blank tape and starts in configuration b, print 0_1 and then halt.

b _ P0,R c
c _ R e
e _ P1 halt

Example 4 demonstrates the use of halt as the final configuration. It follows much the same pattern as Example 3, except that instead of going into an infinite loop, the machine goes into a halt configuration.

The output of a machine running this table of instructions is 0_1, and the head is on square
2
when the machine goes into the halt configuration.

3.2.5 Comments

Any line prefixed with ; is treated as a comment. That is, everything that follows to the right of ; on the line is ignored.

4 Complete (Header and Body) Example

Example 5

Write a header for a blank tape, with a machine starting in configuration b and the head on the first square, and printing output for steps 0 to 200 before halting. Write a body that instructs the machine to print the following pattern:

əə0_0_1_0_1_1_0_1_1_1_0_1_1_1_1_0_1_1_1_1…

Here is code that produces the desired output, adapted minimally from Turing’s paper (Turing 1936, 234, "Example II"):

;; Start with a blank tape
_
;; Start in machine configuration b, with the head on square 0.
;; Print the first 200 steps of the computation.s
b 0 0 200
;; Table of instructions
b _ Pə,R,Pə,R,P0,R,R,P0,L,L o
o 1 R,Px,L,L,L o
o 0 _ q
q * R,R q
q _ P1,L p
p x E,R q
p ə R f
p _ L,L p
f * R,R f
f _ P0,L,L o

Example 5 tells the machine to print 0 followed by a single 1, then 0 followed by two 1s, then a 0 followed by three 1s, and so on. The first line of the body sets up the tape by printing symbols on successive squares, resulting in əə0_0. Then the machine goes into configuration o, and starts the main loop.

Notice that this table of instructions follows Turing’s original very closely, with a few exceptions. In the body, * is used in place of "Any"; _ (underscore) is used in place of "None"; and all spaces have been removed from comma-delimited operations.

The output of a machine running this code is as desired, and the head is on square 38 at step 200.
